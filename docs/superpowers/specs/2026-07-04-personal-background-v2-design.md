# Personal Background v2 - Design

Date: 2026-07-04
Status: Proposed

## 1. Goal

Rebuild `personal-background` as a lean, local-first **kit** that any AI agent can
bootstrap and then use across the user's daily work. The methodology (stable identity,
preferences, constraints, episodes, notes; capture -> distill -> reflect -> apply)
does not change. What changes is delivery and daily-use ergonomics:

- Clone the repo, paste one bootstrap prompt from the README into your own agent, and
  the agent scaffolds the data files and installs the skills + MCP for you.
- Daily use happens **outside this repo**, inside the user's other agent projects,
  through an MCP bridge that can read and append background at any time.

## 2. Principles (unchanged)

- **Local-first**: plain markdown + YAML frontmatter is the only source of truth.
- **Privacy-friendly**: `raw/private/` is gitignored and never leaves the machine.
- **Cross-agent**: works with Factory, Claude, Codex, Cursor, and other compatible agents.
- **Portable markdown**: files stay 100% portable (ASCII slugs, standard YAML frontmatter,
  no tool-specific syntax) so the data home can be synced by git, a cloud folder, or an
  Obsidian vault. The repo itself contains **no** tool-specific integration.
- **Repo files are English**; user data files may be in the user's preferred language.

## 3. Repository as a kit + data home

The repo plays two roles at once:

1. **Kit**: ships the raw material (`kit/`) and a bootstrap prompt that assembles a
   working setup in the user's agent environment.
2. **Data home**: after bootstrap, the repo root holds the user's actual background
   files and is synced as a private git repo (or any other sync layer).

Skills are installed into the user's agent skill directory and point back at the repo
path. The MCP server is registered in the user's agents and also points at the repo path.

### File ownership contract

Because the repo is both a kit and a data home, every path has a clear owner. This is
what makes safe upgrades possible (see section 10).

- **Framework-owned (upstream, upgradeable)**: `kit/`, `methodology.md`, `bootstrap/`,
  `AGENTS.md`, `README.md`, `docs/`, `LICENSE`.
- **User-owned (never overwritten)**: `profile.md`, `preferences.md`, `constraints.md`,
  `episodes/`, `notes/`, `raw/`, `archive/`, `.pbg/`, `index/`.

### Target layout

```
personal-background/
├── README.md               # top section = bootstrap prompt to paste into an agent
├── AGENTS.md               # cross-agent entry point describing how to use the data home
├── methodology.md          # methodology (kept, English)
├── LICENSE / .gitignore
│
├── bootstrap/
│   └── PROMPT.md           # full, canonical bootstrap prompt (README embeds/links this)
│
├── kit/                    # raw material the bootstrap reads to assemble a setup
│   ├── skills/             # canonical skill sources (English), one dir per skill
│   │   ├── personal-profile/SKILL.md
│   │   ├── complete-profile/SKILL.md
│   │   ├── reflect/SKILL.md
│   │   ├── setup-agent/SKILL.md
│   │   ├── localize/SKILL.md      # new: language localization
│   │   └── import/SKILL.md        # new: generic external-source import conventions
│   ├── templates/          # profile/preferences/constraints/episode/note templates
│   ├── config/
│   │   └── settings.example.yml   # preferred language, repo path, index toggle
│   └── manifest.yml        # kit version + changelog (drives the upgrade skill)
│
└── (generated by bootstrap on first run, in the repo root)
    ├── profile.md / preferences.md / constraints.md
    ├── episodes/  notes/  archive/
    ├── raw/{public,private}/
    ├── .pbg/settings.yml   # preferred language, absolute repo path, index toggle
    └── index/              # OPTIONAL, regenerable index output (gitignored)
```

## 4. Bootstrap flow

1. User clones the repo.
2. README top section contains a copy-paste bootstrap prompt (canonical text lives in
   `bootstrap/PROMPT.md`).
3. User pastes it into their agent. The prompt instructs the agent to:
   a. Read `kit/` and `methodology.md`.
   b. Ask the user for a **preferred language** (asked once) and record it, plus the
      absolute repo path, into `.pbg/settings.yml`.
   c. Detect the current agent type and install `kit/skills/*` into that agent's skill
      directory, injecting the absolute repo path so skills operate on the data home.
   d. Register the MCP server in the current agent, configured with the repo path.
   e. Generate empty core files (`profile.md`, `preferences.md`, `constraints.md`) and
      the directory skeleton from `kit/templates/`.
   f. Guide the user to run `complete-profile` or just start talking to capture data.
4. Day-to-day, the user works in other projects and reads/writes background via MCP,
   and runs skills for richer multi-step maintenance.

## 5. Two access surfaces over one data home

### 5.1 MCP server (primary daily bridge)

Registered in the user's agents so that, from **any** project, the agent can read
background and append episodes/notes without opening this repo. Runtime, always on.

Proposed tools:

- Read: `read_profile`, `read_preferences`, `read_constraints`,
  `list_recent(kind, n)`, `search_background(query)`.
- Write: `add_episode(...)`, `add_note(...)`, `append_core(file, section, content)`.
- Behavior: reads/writes in the user's preferred language (from `settings.yml`),
  never reads or writes `raw/private/`, uses `YYYY-MM-DD-slug.md` + frontmatter.
- `search_background` is a grep/scan by default and is the seam where the optional
  index layer plugs in later.

### 5.2 Skills (guided workflows)

Installed into the agent skill directory; for multi-step maintenance:

- `personal-profile`: capture/update background from conversation.
- `complete-profile`: conversational onboarding to fill empty sections.
- `reflect`: consolidate recurring patterns from episodes/notes into core files.
- `setup-agent`: generate agent-specific adapter files from the core.
- `localize`: language localization (see section 6).
- `import`: generic external-source import conventions (see section 7).
- `upgrade`: pull upstream kit updates into the user's repo via reasoned merge (see section 10).

Both surfaces read the repo path from configuration set at bootstrap.

## 6. Language handling

- Canonical repo files are English.
- Preferred language is chosen once at bootstrap and stored in `.pbg/settings.yml`.
- Every skill and MCP write that produces user data uses the preferred language.
- `localize` skill can:
  - generate preferred-language copies of project docs (e.g. `README.zh.md`,
    localized skill descriptions) for reading, leaving the English source untouched;
  - convert the user's own data between languages on request.

## 7. Extensions (conventions only, no concrete integration)

- Define a standard frontmatter contract and an `import` skill import convention: read
  markdown from an external source (e.g. a notes folder or vault), distill into
  `episodes/`/`notes/` following the methodology.
- Reserve the `index/` directory and a settings toggle for future retrieval.
- Do **not** implement any tool-specific integration (no Obsidian-specific code or
  conventions). Obsidian, cloud folders, and git are only example sync/edit choices for
  the portable markdown data home; the repo stays channel-agnostic.

## 8. RAG / graph (optional, regenerable)

- Off by default; controlled by a toggle in `settings.yml`.
- When enabled, a skill/MCP rebuilds `index/` (vector and/or graph) from the markdown;
  `index/` is gitignored and fully regenerable.
- Markdown is always the source of truth; the index is a derived cache.

## 9. Changes relative to current repo

- Keep and adapt: `methodology.md`, the four existing skills, templates.
- Add: `bootstrap/PROMPT.md`, `kit/` layout, `kit/manifest.yml`, `localize` skill,
  `import` skill, `upgrade` skill, `.pbg/settings.yml` concept, a first-class MCP
  server, `index/` seam.
- Remove: `scripts/` (reflect.py, migrate-agent.py) - replaced by skill/MCP-driven flows.
- MCP: elevated from stub to a first-class, required daily bridge.

## 10. Upgrade / evolution skill

The repo is both kit and data home, so a plain `git pull` would entangle upstream kit
changes with the user's data and local customizations. The `upgrade` skill performs a
reasoned merge instead of a code overwrite.

- **User consent is mandatory**: upgrades are always user-initiated (the user runs the
  skill; never automatic or background) and nothing is written until the user explicitly
  approves the merge plan. The user may accept all, accept selectively, or cancel.
- **Upstream source**: a git `upstream` remote pointing at the template repo, or a
  known repo URL the skill fetches from. The user's own private remote stays separate.
- **Version awareness**: `kit/manifest.yml` carries a kit version and changelog; the
  skill compares the user's current kit version to upstream and shows a readable summary
  of what changed.
- **Scope**: only framework-owned paths (section 3 ownership contract) are ever
  modified. User-owned paths are never touched.
- **Flow**:
  1. Determine upstream source and fetch the latest kit.
  2. Diff upstream kit vs. the user's current kit and installed skills.
  3. Produce a merge plan: list added, changed, and user-customized items separately;
     show conflicts side by side.
  4. Wait for explicit confirmation.
  5. Apply the merge (three-way where the user customized a framework file), re-install
     updated skills into the agent skill directory, and update the MCP registration if
     its interface changed.
  6. Record the new kit version in `.pbg/settings.yml`.
- **Guarantee**: never blind-overwrites; user data and customizations are preserved or
  surfaced as conflicts for the user to resolve.

## 11. Out of scope

- Any concrete third-party integration (Obsidian plugins, sync services).
- Shipping a populated index or a required vector store.
- Bulk-importing raw chat logs without distillation.
